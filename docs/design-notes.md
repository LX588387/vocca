# 设计笔记

记录几个关键取舍与「为什么这么做」，方便后来者（和未来的我）理解。

## 为什么核心是纯 numpy？

音频生成模型的「难」通常在训练和声码器，而框架层真正需要讲清楚的是**机制**：
条件如何注入、CFG 如何组合、KV cache 如何保证增量与整段一致、流式如何与离线
对齐。这些用纯 numpy 就能完整、确定地表达，还能在没有 GPU 的 CI 上秒级跑完。
torch 只作为可选后端，用来交叉验证前向、并为将来接训练留口子。

这也和我音频系列前几个项目的取舍一致：先把接口和不变量做对，再谈规模。

## 为什么随机初始化权重也「有用」？

参考实现默认不加载训练权重，产出的是结构化噪声而非自然语音。但框架的价值不
依赖权重质量：

- cache 一致性、流式一致性、CFG 数学、采样正确性——都能在随机权重下严格断言。
- 换上真实 `.npz` 权重后，同一套代码即可产出真实语音，接口不变。

这与 auris 里 `EchoBackend` 的思路一脉相承：**先证明链路打通，再谈内容质量**。

## KV cache 与「绝对位置」

要让 `prefill + step` 与整段 `forward` 数值一致，位置编码必须用**绝对位置**、
因果掩码也按绝对位置构造。RoPE 天然满足：位置只进不退，缓存里的 K/V 早已带上
各自位置的旋转，新 query 只需按自己的绝对位置旋转即可。于是「一次算 N 个」与
「算 N 次各 1 个」得到逐元素相同的结果（浮点误差 ~1e-7）。

## 流式为什么能做到「逐样本相等」

关键在编解码器接口 `decode_chunk(tokens, state)`：把跨块的相位、上一帧幅度等
放进显式 `state` 里携带。整段 `decode` 只是「用初始 state 调一次 decode_chunk」，
而流式是「分多次调，state 顺着传」。两者走的是同一段合成代码，因此拼接结果与
整段结果 bit 级一致。相位连续、幅度帧内线性过渡，既避免爆音，又不破坏这条等式。

## 条件为什么用「前缀」而不是逐层 FiLM？

前缀条件（把控制向量当成序列开头的若干 token）实现最简单，且天然兼容 KV cache
与 CFG：空条件就是「零前缀」，条件与空条件只差前缀那几列。逐层 FiLM / adaLN 表达
力更强，但会让 cache 与 CFG 的记账复杂不少。作为参考框架，前缀是更清晰的选择；
`ControlEncoder` 已经把「多前缀 token」的口子留好了。

## 编解码器：振荡器而非神经声码器

默认 `OscillatorCodec` 用五声音阶做加性合成，好处是**可听 + 确定 + 流式精确**，
且零依赖。它显然不是高保真声码器——真实项目里应当换成 soniq 那样的 RVQ 解码器
或神经 vocoder，只要实现 `Codec` 接口即可，其余代码无需改动。

## 已知的小瑕疵

- `OscillatorCodec.encode` 只按主频匹配，是有损的近似逆，仅供往返测试。
- `stopped_by_eos` 在「恰好用满 `max_new_tokens`」时无法与「采到 EOS」区分——
  这对参考实现无关紧要，暂不细究。
- 采样里的 top-p 在极端温度下可能只保留一个 token；这是刻意的（至少留一个）。
